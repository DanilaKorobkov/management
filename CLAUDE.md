# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## О репозитории

Репозиторий содержит управленческую документацию команды — процессы, регламенты, чеклисты. Это **не кодовый проект**, а база знаний по инженерному менеджменту.

Язык документации — русский. Все документы пишутся на русском языке.

## Структура

```
processes/           — описания процессов команды (публикуются в Confluence)
retro.private/       — приватные заметки к ретро (не публикуются)
.claude/
  confluence-sync.json — маппинг файлов на Confluence-страницы
  skills/sync-confluence/ — skill для синхронизации с Confluence
```

### Соглашение о private

Файлы и папки с `private` в любой части пути **не публикуются** в Confluence. Примеры:
- `retro.private/` — вся папка приватная
- `processes/feature-requests.private.md` — черновик процесса, не готов к публикации

Чтобы опубликовать приватный файл — убери `private` из имени.

## Синхронизация с Confluence

Процессы из `processes/*.md` публикуются в Confluence (пространство MIS) через skill `/sync-confluence`.

- **Confluence-пространство:** MIS (Shared Services)
- **Родительская страница:** "Процессы" (ID `842230758`)
- **Корневая страница:** "It Platform Dev/Integrations" (ID `842230756`)
- **Структура:** плоская — контентные страницы прямо под "Процессы"
- **Название страницы** берётся из первого `# H1` заголовка markdown-файла (markdown — source of truth)
- **Формат контента:** markdown конвертируется в Confluence storage XHTML (макрос markdown недоступен на этом инстансе)
- **Маппинг и версии:** `.claude/confluence-sync.json`
- **Конфликты:** если кто-то отредактировал страницу в CF вручную — skill обнаружит по номеру версии и спросит перед перезаписью
- **Подтверждение:** перед любыми изменениями в CF skill показывает список действий и ждёт подтверждения

Skill вызывается **только явно**: `/sync-confluence` или `/sync-confluence <filename>`.

## Команда

- 6 инженеров, выделенного продакт-менеджера нет
- Решения по приоритетам (техдолг, feature-request'ы) принимают инженеры самостоятельно, без участия менеджера

## Ключевые процессы

### Управление техдолгом (`processes/tech-debt.md`)

- Квота 20% ёмкости спринта на плановый техдолг (Major/Normal/Minor), зарезервирована до добавления остальных задач
- Critical-задачи идут **вне квоты** — не считаются в 20%
- Общий лимит на весь техдолг (Critical + квота): **50%** ёмкости спринта. Причина: компания ожидает видимый результат для пользователей каждый спринт
- Приоритеты техдолга определяют инженеры, не менеджеры. Автор задачи ставит оценку (SP) и приоритет самостоятельно при создании
- Приоритеты: Critical → Major → Normal → Minor
- Правило 30 минут: если починка < 30 мин и не ломает скоуп — чини сразу без тикета
- Принцип "попутного техдолга": если в спринте есть фича в модуле X — посмотри, есть ли техдолг в модуле X
- Если > 13 SP — декомпозируй: родительский Tech-task (0 SP) + дочерние (≤ 13 SP каждый)
- Техдолг-ревью: раз в 2 недели, 15-30 мин, в середине спринта, фасилитирует скрам-мастер
- Тикет старше 6 спринтов — ревалидация на техдолг-ревью
- Целевые метрики: ≤30 открытых задач, суммарный вес < 100% velocity, среднее время жизни ≤ 3 спринта, время решения p50 ≤ 1 спринт, p90 ≤ 3 спринта

### Разбор feature-request'ов (`processes/feature-requests.private.md`)

Процесс пока в статусе **private** (черновик, не опубликован в CF).

- Единая точка входа — PrioPlan (все идеи из чатов и личных сообщений перенаправляются туда)
- Разбор идей: раз в 2 недели, до 30 мин, на второй неделе спринта (чтобы принятые идеи могли пойти в следующий спринт)
- Фасилитатор: скрам-мастер или замещающий. Участники: все инженеры, без менеджера
- Дежурный по идеям (в рамках недельной ротации): 2 раза в неделю по 5-10 мин проверяет новые идеи на дубликаты, перенаправляет пользователей из чатов в PrioPlan, отмечает неполные идеи
- Голоса пользователей в PrioPlan влияют на приоритет: при прочих равных побеждает идея с большим количеством голосов
- Факторы решения: голоса в PrioPlan, импакт для продукта, описание проблемы, стоимость реализации (SP), срочность
- Статусы в PrioPlan: Новая → Требует уточнений / На рассмотрении / Приняты / Отклонены → В работе → Сделаны
- Принятая идея → Feature-тикет в Jira через "+New" в PrioPlan. Небольшая идея реализуется в самом Feature-тикете; крупная — Feature + связанные задачи (caused by): UserStory / Development / Tech-task
- Отклонённые идеи получают комментарий с объяснением в PrioPlan
- Идея "Требует уточнений" > 4 недель без ответа — закрывается. Принятая идея > 3 спринтов без движения — пересмотр
- Статусы "В работе" и "Сделаны" синхронизируются автоматически через Status Auto-Update Rules (Jira — мастер-система)

## Соглашения при редактировании

- Документы используют Markdown с таблицами, блок-схемами в code-блоках и FAQ-секциями
- Структура документа: цель → принцип → правила → примеры → метрики → чеклист → FAQ
- Примеры должны быть конкретными (с названиями сервисов, SP, файлами)
- Пограничные случаи описываются явно
- Первая строка `# H1` определяет название страницы в Confluence
